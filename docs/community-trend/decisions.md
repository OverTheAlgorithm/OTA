## D-001. 일일 카운트 = "신규 관측분만"(신규유입형), carryover 재카운트 안 함

- **결정**: `ct_tag_daily.post_count`는 그날 **처음 관측된** 항목만 센다. 베스트판에 며칠 머무는 글(carryover)은 첫날만 카운트, 이후 스킵.
- **맥락**: 같은 글이 베스트판에 여러 날 떠있음. "오늘 새로 뜬 글만 셀까(신규유입형)" vs "떠있는 날마다 셀까(재고형)".
- **이유**:
  1. 원래 `ct_seen_posts` 지문은 **AI 재태깅 토큰 낭비 방지**용이었는데, 이미 본 글을 스킵하면 자연히 신규유입형이 된다 (두 요구가 한 메커니즘으로 해결).
  2. 재고형을 하려면 글별 태그 매핑을 저장해 매일 재집계해야 함 → **"집계만 저장(본문/글단위 미저장)" 원칙이 깨짐**. 저작권 가드레일과 충돌.
  3. 트렌드 신호로 "오늘 새로 끓은 양"이 "재고 존재감"보다 깔끔하고 의미 있음.
- **트레이드오프**: "지금 베스트에 이 주제가 총 몇 건 떠있나"(누적 존재감)는 표현 불가. 수용함.
- **버린 대안**: 재고형. (집계만 저장 원칙과 양립 불가라 탈락)

## D-002. 보수적 임계값 및 필터링 시점 = 환경변수 `CT_MIN_TAG_COUNT`(기본 3), 노출 단계 필터

- **결정**:
  1. **AI의 중립적 추출**: AI는 게시글의 댓글/추천 수 정보 없이 오직 본문의 텍스트만 보고 중립적으로 모든 태그를 매핑하여 추출합니다. AI 단계에서 임계치 필터를 적용하지 않습니다. (단, 새로운 태그를 신규 제안하는 과정에선 보수성 지침을 유지합니다.)
  2. **정적 가중치 계산**: 매핑된 게시글들의 추천/댓글 수를 바탕으로 백엔드에서 로그 스케일 가중치를 산출하여 합산합니다.
  3. **노출 단계 필터**: 최종 합산된 점수가 환경변수 `CT_MIN_TAG_COUNT`(실제로는 최소 점수/가중치 임계치) 미만인 태그는 일반 사용자 웹(차트/배달) 노출 시에만 숨김 처리합니다. 관리자 워크시트 화면에는 무조건 모든 태그가 노출됩니다.
- **맥락**: AI 단계에서 보수적으로 거르게 할 경우 데이터 수집의 중립성이 훼손되고 어드민의 제안 범위가 좁아집니다. 또한, 가중치 점수 체계 도입으로 단순 개수(Count)가 아닌 가중 점수합(Score/Weight)이 임계 기준이 됩니다.
- **이유**:
  1. 저장 단계나 AI 추출 단계에서 임계치로 잘라버리면 나중에 임계값을 바꿀 때 **재수집 불가**. raw 데이터(1건 매핑이라도)를 다 살려서 suggestions/DB에 저장해 두어야 사후 설정 튜닝이 가능합니다.
  2. 관리자는 시스템이 인지한 모든 주제와 가중치 점수를 보고 편집·확정해야 합니다.
- **트레이드오프**: `CT_MIN_TAG_COUNT` 환경변수명에 'Count'가 들어가 있어 점수 임계치(Score Threshold) 개념과 명칭상의 괴리가 다소 있으나, 호환성을 위해 유지합니다.
- **버린 대안**: AI 프롬프트에 `MinCount` 필터를 전가해 AI 단계에서 보수적으로 폐기하는 방안 (어드민에 안 보여 튜닝이 불가능하여 탈락).

## D-003. 태그 = 공통 풀 하나, 메타/주제는 부착 위치로 구분, 겸용 허용

- **결정**: 태그 리소스는 단일 풀(`ct_tags`). 같은 태그가 ① 커뮤니티 영구 성향(메타, `ct_community_tags`) ② 일일 논의 주제(`ct_tag_daily`) 어느 쪽으로도 부착될 수 있다. 한 태그의 메타·주제 겸용을 막지 않는다.
- **맥락**: '남성향'(메타)과 '남성 인권'(주제)을 같은 풀에 둘지, 별도 풀로 나눌지.
- **이유**:
  1. "태그 리소스 자체는 공통"이라는 합의. 역할은 *붙는 자리*로 충분히 구분됨.
  2. 메타/주제를 별도 테이블·별도 풀로 쪼개면 중복·동기화 부담. 부착 테이블 분리로 이미 역할이 갈림.
  3. 과한 제약(겸용 금지)은 YAGNI. 실제 충돌 생기면 그때 제약 추가.
- **트레이드오프**: 한 풀에 성향태그와 주제태그가 섞여 보임. 축(axis)으로 묶어 완화.
- **버린 대안**: 메타 태그 / 주제 태그 별도 풀 2개. (동기화 부담, 불필요한 분리)

## D-004. 첫 4개 사이트 = dogdrip · clien(자동) / fmkorea · 더쿠(수동)

- **결정**: 초기 4개. **자동 경로** dogdrip·clien, **수동 경로** fmkorea·더쿠.
- **맥락**: 후보 다수 중 첫 스타트 4개 선정. 성향 강하고 인구 충분한 곳 우선.
- **이유**:
  1. **robots 실측**(2026-06-24): dogdrip(Allow `/`, crawl-delay 10), clien(`/service/board/` Allow) = 자동 가능. fmkorea(HTTP 429 Cloudflare), 인스티즈(403), 더쿠(robots 404·CDN 차단 의심) = 자동 난망 → 수동.
  2. 자동·수동 **두 경로를 1일차부터 검증**(설계상 둘 다 핵심).
  3. 성향 분산: 남초2(dogdrip·fmkorea) · 여초1(더쿠) · 진보1(clien).
- **역설 기록**: 성향 제일 센 곳(fmkorea, 인스티즈)이 anti-bot으로 크롤 제일 어려움 → "막히면 사람이 입력" 설계가 1일차부터 정당화됨.
- **버린 대안**: 일베(평판·법적 PR 리스크), 소규모·전적 게시판류(의견추출 약함).

## D-005. 본문·제목 미저장, "집계만 저장"

- **결정**: 글 본문·제목은 어떤 테이블에도 저장 안 함. 수집된 글의 본문/제목은 AI 프롬프트를 전송하고 응답을 받는 짧은 실행 시간 동안에만 메모리에 존재하며 영구히 저장되지 않습니다. 중복검정은 역산 불가 해시 지문만 사용합니다. AI 제안 상세 정보(태그/밈 제안 및 지문 목록)는 Redis 캐시(TTL 48시간)에 임시 보관되며, 어드민 승인 후에는 오직 지문 해시(Fingerprint)만 DB `ct_seen_posts`에 기록되고 제안 정보는 파기됩니다.
- **이유**: 저작권법은 *표현*을 보호하지 *사실·트렌드*는 아님. 본문 미보유 + 미인용 + 미표시 → 저작권 침해 성립 어려움. 파생 집계(태그 수)만 산출.
- **참고**: 상세 법적 논거는 `방향성-기획.html` §4.

## D-006. robots Disallow = 우회 안 함, 사람이 대신 입력

- **결정**: robots가 대표 게시판을 불허하면 그 사이트는 **자동 접근 0**. 사람이 직접 브라우징해 태그 수동 입력. robots.txt 자체는 매일 수집·전이 추적.
- **이유**:
  1. robots 무시 자체는 형사 위법 아니나, "서버 예정 접근범위 초과"의 결정적 증거가 됨 (정보통신망법 — 여기어때 v 야놀자 대법 판례). 최선의 방어를 스스로 버리는 꼴.
  2. 사람의 웹 열람은 합법. 크롤링 0 → 정보통신망법·robots 문제 원천 소멸.
  3. robots.txt는 "읽으라고 존재"하는 파일이라 수집 자체는 보편 허용.
- **트레이드오프**: 막힌 사이트는 사람 손이 매일 필요. 베스트글 적어(10~20분) 감당 가능 판단.

## D-007. AI 제안 + 사람 확정 필수 (자동확정 없음)

- **결정**: AI는 태그를 *제안*만. 사람이 확정해야 DB(`ct_tag_daily`) 반영. 신규 태그·신규 밈후보도 사람 게이트 필수. (단 *확정밈* 매칭 카운트는 AI 자동.)
- **이유**: 태그를 보수적으로 평가해야 함(스펙 2번). AI 단독은 과잉 태깅 위험. 작업량 적어 사람 검수 감당 가능. 정/부 2인 당번(이중승인 아님, 단일클릭 확정).

- **D-008. `ct_collection_runs` 테이블 없음** — "런"=달력 날짜라 군더더기. 모든 것을 `stat_date` + `ct_worksheets(community, date)`로 키잉.
- **D-009. AI 호출 1패스 통합** — 태그제안 + 밈매칭 + 밈후보를 호출 1회로. 입력(제목 묶음) 동일하니 합치는 게 자연스럽고 비용 1/3.
- **D-010. 코호트 = 메타태그 파생 쿼리, 별도 테이블 없음** — `ct_cohorts`/`ct_community_cohorts` 폐기. "남초"=메타태그 '남성향' 붙은 커뮤니티 합산. v1 코호트=단일 메타태그(복합조건은 향후).
- **D-011. 밈 확정 삭제 = retire(status), hard delete 아님** — 과거 `ct_meme_daily` 카운트 보존 위해. 밈은 짧게 살지만 트렌드 이력은 남겨야 함.
- **D-012. 밈 후보 삭제 = 영구거부(blacklist)** — 단순하게. AI가 같은 표현 재등록 못 하게 차단.

## 아키텍처 정비 결정 (2026-06-28 반영)

## D-013. 커뮤니티 트렌드 수집 파이프라인의 데이터 흐름 및 역할 분담

- **결정**: 수집 파이프라인(`Pipeline.RunDaily`)은 다음 구조적 단계들을 명확히 따르며, 개별 어댑터(`Adapter`), AI 분석(`Tagger`), 그리고 시스템 가중치 계산 간의 역할을 엄격히 분리합니다.
- **역할 분담 및 흐름**:
  1. **어댑터 fetch**: 각 커뮤니티 어댑터는 최근 게시글 목록(TextUnit, SourceID, comments, upvotes)을 가져옵니다.
  2. **robots.txt 검문 (robots gate)**: 허용된 경로만 자동 수집을 진행하며, Disallow된 경우 즉시 수동 입력(manual) 워크시트를 생성하고 대기합니다.
  3. **중복 필터링 (Deduplication)**: `SourceID` 기반의 해시 지문이 `ct_seen_posts`에 존재하는지 비교하여 **신규 관측글(fresh items)만 선별**합니다.
  4. **AI 분석 (Tagger)**: 신규 관측글들의 **순수 제목 텍스트만**을 Gemini에 전달하여 중립적으로 기존 분류 체계에 태깅(`post_indices`)하고 신규 밈 후보를 추출합니다. AI 호출 시 댓글/추천 수 등은 전달하지 않으며, AI는 가중치 계산이나 임계 필터링 역할을 수행하지 않습니다. (신규 태그 제안 시에만 매우 보수적으로 제안하도록 제한합니다.)
  5. **가중치 점수 산출 (Go Backend)**: AI가 리턴한 태그 매핑 인덱스(`post_indices`)를 바탕으로, 백엔드가 원본 게시글의 댓글/추천수를 역조회하여 정해진 가중치 공식(`1.0 + math.Log10(1.0+val)`)에 따라 **태그 점수를 결정론적(deterministic)으로 계산**합니다.
  6. **임시 보관 (Redis)**: 계산된 태그 점수 및 밈 카운트, 글의 지문 목록을 Redis 임시 저장소(`CTSuggestionStore`)에 48시간 동안 임시 보관(TTL)합니다.
  7. **어드민 컨펌 및 DB 영구화**: 관리자가 검토 후 컨펌을 누르면, 임시 저장된 지문 해시들을 `ct_seen_posts`로 밀어 넣어 중복 처리를 완료하고, 태그 점수들을 `ct_tag_daily` 및 `ct_community_daily`에 영구적으로 인서트/업데이트(Upsert)합니다.
- **이유**:
  1. 새로운 커뮤니티 어댑터를 추가할 때 어댑터는 단순 최근 글 목록 수집 기능만 구현하면 되고, AI 프롬프트나 가중치 산출 공식을 전혀 몰라도 되므로 **관심사가 완전히 분리**됩니다.
  2. 저작권 이슈를 피하기 위해 글의 본문/제목은 DB에 들어가지 않고, 오직 실시간 API 전송 및 Redis 지문 해시 변환용으로만 사용된 뒤 완전히 소멸합니다.
