커뮤니티의 "진짜 관심사"를
읽어내는 서비스

각 커뮤니티가 어떤 논제에 끌리는지, 그게 어제·지난주보다 얼마나 뜨거워졌는지를 데이터로 만든다. 글을 베끼는 게 아니라, 분위기를 측정한다.

기존 WizLetter 수집 기능과 완전히 분리된 신규 피쳐 · 표현(그래프)은 별도 개발

1무엇을 만드나

한 문장: "이 커뮤니티는 요즘 이런 얘기에 빠져있다"를 매일 자동으로 알아낸다.

📡 비유하자면 — 우리는 커뮤니티 글을 읽어서 옮기는 기자가 아니다. 각 동네 위를 떠다니며 "오늘 이 동네는 무슨 얘기로 떠들썩한가"를 기록하는 기상관측소에 가깝다. 구름의 내용물(글 본문)이 아니라, 날씨(논제의 흐름)만 잰다.

예를 들어 어떤 남초 사이트에 "여성의 무고 사건" 글이 자주 올라온다면 → 우리는 그 글들을 인용하지 않는다. 대신 젠더축 남성 인권 태그가 그날 몇 번 다뤄졌나만 센다.

2왜 가치 있나

🧭 여론의 지형도

커뮤니티마다 추종하는 주제가 뚜렷하다. 그 지형을 한눈에 본다.

📈 변화 감지

"이 주제가 갑자기 끓어오른다"를 어제·지난주 대비로 포착한다.

👥 집단 비교

남초 vs 여초, 2030 vs 4050의 관심사 격차를 비교한다.

단일 사이트 통계보다 집단(코호트) 단위의 인사이트가 훨씬 강력하다. "남초는 지금 무엇에 분노하는가" 같은 질문에 데이터로 답할 수 있다.

3어떻게 동작하나

하루에 딱 한 번. 사이트의 대표 게시판(베스트)에서 최근 1일 글만 본다. 전수조사 아님.

🔍

① 관측

크롤링 허용 사이트는 자동으로 베스트글 제목·반응만 가져온다. 본문은 안 가져온다. 이미 본 글은 건너뛴다(중복 검정).

🤖

② AI 제안

AI가 제목들을 보고 "이건 어떤 논제다" 태그를 제안한다. 기존 태그 우선, 정말 없을 때만 새 태그 제안.

✅

③ 사람 확정

사람이 같은 화면에서 AI 제안을 검수·수정하고 최종 확정한다. 확정해야 DB에 들어간다. (품질을 보수적으로 유지하는 안전장치)

📊

④ 집계

확정되면 (사이트 × 태그 × 날짜 = 글 개수)만 쌓는다. 제목은 폐기. 이 숫자로 증감·집단비교가 나온다.

🔁 자동이 막히면 사람이 메운다

크롤링이 막힌(robots 불허) 사이트는 자동 접근을 아예 안 한다. 대신 사람이 직접 그 사이트를 눈으로 둘러보고(브라우징은 누구나 하는 일) 같은 화면에 태그를 입력한다. 하루 베스트글은 의외로 적어서 10~20분이면 충분하다. AI가 하든 사람이 하든 나가는 데이터 모양은 똑같다.

4가장 중요한 질문: 이거 합법이야?

팀이 가장 불안해할 부분. 결론부터.

⚖️
본문을 절대 만지지 않고 집계 숫자만 만들면, 저작권 분쟁 소지는 사실상 없다. 다만 안전은 "무엇을 저장하나"가 아니라 "어떻게 긁나"로 지킨다.

왜 저작권은 걱정 안 해도 되나

저작권법은 표현을 보호하지, 사실·아이디어·트렌드는 보호하지 않는다. "이 사이트에서 이런 논제가 많다"는 사실의 관찰이다. 글의 문장을 복제하지 않고, 인용하지 않고, 보여주지도 않으니 보호 대상 자체가 우리 손에 없다.

그래도 남는 3가지 (저작권과 다른 종류의) 리스크

이건 본문 저장 여부와 무관하게 존재한다. 그래서 수집 방식으로 막는다.

중간

DB제작자의 권리 (저작권법 §93)

게시판도 일종의 "데이터베이스"라, 상당 부분을 체계적으로 퍼가면 권리 침해가 될 수 있다.

→ 회피: 목록 자체를 복제하지 않고 파생 집계(태그 수)만 만든다.

중간

정보통신망법 — 실제 형사 판례 있음

"여기어때 vs 야놀자" 대법원 판결: 공개된 데이터라도 서버가 예정한 범위를 넘는 크롤링은 침입죄가 됐다. 핵심은 방법 — 과도한 트래픽, 접근통제 우회.

→ 회피: 하루 1회, 베스트판만, 정직한 봇 신원, robots 불허 시 자동접근 0.

중간

부정경쟁방지법 — "성과 도용"

남의 노력 결과물에 무임승차하는 것. 가장 현대적인 회색지대.

→ 회피: 원본을 대체하지 않는 변형된 분석(집단 트렌드)을 만든다. 특정 사이트 데이터가 아니라 "남초 전체" 같은 집계라 한 회사의 성과로 환원되지 않는다.

robots.txt — 우회가 "정당한 우회"인 이유

robots.txt 자체는 법이 아니다. 하지만 그게 "불허"라고 적혀 있는데 무시하면, 상대가 "예정 범위를 넘었다"고 입증하기 쉬워진다. 우리의 최선의 방어("공개 데이터였다")를 스스로 버리는 꼴이다.

그래서 우리의 우회는 "몰래 뚫는 우회"가 아니라 "사람이 대신하는 우회"다:

막혀 있으면 → 기계는 손 뗀다 → 사람이 직접 눈으로 본다.
사람이 웹사이트를 브라우징하는 건 누구나 하는 합법 행위다. 크롤링이 0이면 정보통신망법·robots 문제가 원천적으로 발생하지 않는다. 이게 회색지대를 깔끔하게 비켜가는 핵심 설계다.

🔎 robots 상태를 매일 추적하는 이유

robots.txt는 "읽으라고 만든" 파일이라 그것을 가져오는 것 자체는 보편적으로 허용된다. 그래서 모든 대상 사이트의 robots를 매일 확인하고, 허용↔불허가 바뀌면 관리자 화면에 보여준다.

덕분에 "어제는 허용이라 자동으로 돌렸는데 오늘 막혔네 → 오늘부터 사람이 입력" 같은 전환을 놓치지 않는다. 합법성의 근거를 시점별로 남기는 장부 역할도 한다.

5우리가 저장하는 것 / 절대 안 하는 것

이 표 하나가 합법성의 절반이다. 팀에게 가장 안심되는 그림.

✅ 저장한다

  • 사이트 × 태그 × 날짜 = 글 개수
  • 그날 베스트판 총 글 수 (점유율 계산용)
  • 태그 분류체계 (축 + 태그 이름)
  • 중복 방지용 되돌릴 수 없는 해시 지문
  • robots 상태 변화 이력

🚫 절대 저장 안 한다

  • 글 본문 (아예 안 가져오거나 즉시 폐기)
  • 글 제목 (작업 중 잠깐 떠 있다 폐기)
  • 원본 글의 인용·요약·재배포
  • 작성자 정보
  • 되돌릴 수 있는 형태의 어떤 원문도

6핵심 결정과 그 이유

논의 끝에 합의한 방향들. "왜 이렇게 정했나"를 같이 적는다.

👥

사이트별이 아니라 "집단(코호트)"으로 묶어 보여준다

남초 / 여초 / 40대+ / 30대- 같은 군집 통계. (한 사이트는 여러 집단에 동시 소속 가능 — fmkorea = 남초 + 30대-)

왜? ① 법적으로 더 안전 — "특정 회사 데이터"가 아니라 집계라 성과 도용 주장이 약해진다. ② 제품적으로 더 강력 — 집단 비교가 단일 사이트보다 가치 높다. (데이터는 사이트별로 정밀하게 쌓고, 보여줄 때만 묶는다)

🤖

AI가 제안하고, 사람이 반드시 확정한다

AI 자동확정이 아니라, 사람 검수를 거쳐야 DB에 들어간다.

왜? 태그를 보수적으로 평가해야 하기 때문. AI 단독은 과잉 태깅 위험. 하루 작업량이 적어(10~20분) 사람이 충분히 감당. AI와 사람이 정확히 같은 화면에서 일한다.

🏷️

태그는 "축 + 태그" 2계층, 그리고 정밀하게

예: 젠더축 남성 인권 정치축 우파 지지

왜? "우파"가 아니라 "우파 지지"처럼 입장을 정확히 표현해야 오해가 없다. 축으로 묶으면 대시보드에서 그룹·필터가 쉽다. 기존 태그 우선, 진짜 없을 때만 AI가 새로 만든다.

📐

측정 단위는 "글 개수 + 일일 총량"

태그별 글 수와 그날 전체 글 수를 같이 저장. 증감은 저장하지 않고 필요할 때 계산.

왜? 단순하고 견고하다. 총량을 같이 들고 있어 나중에 "점유율"도 계산 가능. 베스트판 볼륨이 날마다 달라도 보정된다.

🧱

사이트마다 전용 어댑터, 분석은 정적+동적 혼합

각 커뮤니티 구조에 맞는 어댑터로 정해진 방식대로 데이터를 모으고(정적), 태그 판단만 AI(동적)에 맡긴다.

왜? 데이터 준비 과정이 예측 가능·재현 가능해야 디버깅·신뢰가 된다. 창의적 판단이 필요한 곳에만 AI를 쓴다.

7이번 범위 (선 긋기)

✅ 이번에 만든다

데이터 수집 → 태깅(AI+사람) → 집계 저장 → 그래프로 그릴 수 있는 형태의 데이터까지. 즉 "재료 준비" 완성.

⏸️ 이번엔 안 한다

최종 결과물을 가공해 보여주는 화면/그래프는 별도 개발. 기존 WizLetter 기능과도 완전히 분리해서 따로 간다.

8팀에게 묻고 싶은 것

방향에 합의가 필요하거나, 아직 안 정한 것들.

대상 사이트 목록. 더쿠·fmkorea·dogdrip·mlbpark·clien 외에 "충분한 영향력"의 기준은? 초기엔 몇 개로 시작할까?
코호트 정의. 남초/여초/2030/4050 외에 더 필요한 집단이 있나? 사이트→집단 분류는 누가 책임지나?
태그 축의 초기 set. 젠더·정치·연령 말고 어떤 축을 미리 깔아둘까? (사회·경제·연예 등)
사람 확정 운영. 매일 누가 태깅 검수를 도나? 1인? 당번제? 작업 부담 실측 필요.
"보수적 태깅"의 합의된 기준. 몇 건 이상 다뤄져야 태그로 인정? 애매할 때 원칙은?
커뮤니티 트렌드 · 방향성 기획 초안 · 팀 논의 후 정식 스펙으로 확정 예정
본 문서는 결정의 "이유"를 공유하기 위한 것 — 구현 세부는 별도 기술 스펙에서 다룬다.
