# 체류 시간 검증 고도화 (Layer 1.5) 도입 필요성 및 설계 문서

## 1. 개요 및 논의 배경
OTA 서비스는 사용자가 특정 주제(기사 등) 페이지에 진입하여 일정 시간(N초) 이상 체류했을 때 코인(환금성 보상)을 지급하는 구조를 가지고 있습니다.
현재 백엔드에서는 `init-earn` 시점에 캐시에 기록을 남기고, `earn` 요청 시점에 캐시에 기록된 시작 시간과 현재 시간을 비교하여 N초 이상이 경과했는지를 **강제**합니다. (시스템 시간 스킵 불가)

하지만 이 방식에는 프론트엔드 환경의 근본적인 취약점이 존재합니다:
- **백그라운드 탭 어뷰징**: 사용자가 `/topic/:id` 페이지를 열어두기만 하고(init-earn 호출), 곧바로 다른 탭으로 이동하거나 창을 내려 백그라운드로 전환하더라도, 백엔드는 사용자가 실제로 화면을 보고 있었는지(Active Tab) 여부를 알 수 없습니다.
- **JS 변조 공격**: 프론트엔드의 타이머 로직(JS)을 우회하여, `init-earn`을 보낸 뒤 백그라운드에서 스크립트만으로 10초를 대기하고 `earn` API를 전송하는 꼼수가 가능합니다. (Cloudflare Turnstile을 도입하더라도, 해당 시점에 봇이 아님을 증명할 뿐 10초 내내 탭을 켜두었는지는 보장하지 못함)

이러한 **실제 활성 브라우저 탭에서의 온전한 체류 시간**을 기술적으로 강제하기 위해 논의된 것이 바로 **Layer 1.5 (상태 공유 기반 Ping 검증)** 입니다.

## 2. Layer 1.5 방어 논리 (어떻게 막는가?)
프론트엔드와 백엔드가 "활성 상태"에 대한 핑(Ping)을 주기적으로 주고받으며 체류 시간을 증명합니다.

1. **requestAnimationFrame (rAF)의 특성 활용**: 
   프론트엔드 브라우저는 현재 탭이 백그라운드로 가거나 안 보이는 상태가 되면 `requestAnimationFrame`의 호출 주기를 극단적으로 늦춥니다(또는 멈춤). 이를 이용해 rAF 안에서 타이머가 틱(Tick)을 발생시키도록 합니다.
2. **Ping 릴레이 (상태 증명)**:
   단순히 로컬에서만 틱을 세는 것이 아니라, `init-earn` 시점에 서버가 "2초마다 서명(Nonce/Token)을 담아 Ping을 보내라"고 지시합니다.
3. **서버의 엄격한 검증**:
   서버는 각 유저의 `init-earn` 세션에 대해 `[2초, 4초, 6초, 8초, 10초]` 시점마다 Ping이 정확하게 들어오는지 기록합니다. 
   - Ping이 너무 늦게 오면 ➡️ **"탭을 백그라운드로 내렸구나"** -> 획득 실패 처리
   - Ping이 한꺼번에 오면 ➡️ **"스크립트로 조작했구나"** -> 획득 실패 처리

## 3. 도입 (구현) 방법 제안

### 3.1. 백엔드 (Server) 수정
1. `/api/v1/level/init-earn` 응답에 세션(Session) ID와 Ping 주기(예: 2초)를 포함시킵니다.
2. `POST /api/v1/level/ping` 엔드포인트를 신설하여 프론트의 Ping을 수신합니다.
   - Redis(또는 현 Otter 캐시)에 해당 세션의 Ping 수신 횟수와 타임스탬프를 갱신합니다.
3. 최종 `/api/v1/level/earn` 호출 시, 
   - 경과 시간 N초 확인은 물론, **Ping이 N초/2초 번만큼 정확한 인터벌로 수신되었는지**를 추가 검증합니다. 통과해야만 코인을 지급합니다.

### 3.2. 프론트엔드 (Web) 수정
1. `setInterval` 대신 `requestAnimationFrame` 기반의 루프를 생성합니다.
2. 루프 내부에서 로컬 시간을 측정하여, 정확히 2초가 경과할 때마다 백엔드의 `/ping` API를 비동기로 호출합니다.
3. `document.visibilityState !== 'visible'` 일 경우 루프를 정지하거나 즉시 코인 획득 실패 처리를 띄우도록 합니다.

## 4. 결론
Layer 1.5는 구현 복잡도가 상승하고(상태 관리, 네트워크 트래픽 증가) 유저의 디바이스 성능 스파이크 등에 다소 민감해질 수 있다는 단점이 있습니다. 
하지만 환금성 서비스로서 **"유저의 확실한 Active 상태 체류"**를 강제해야 하는 시점이 온다면, 단순히 클라이언트의 JS 방어를 넘어 서버가 주도적으로 활성 뷰(View)를 채점하는 이 아키텍처가 필수불가결합니다. 
서비스의 초기 트래픽과 어뷰징 비율을 모니터링한 후, 매크로 등 백그라운드 파밍 빈도가 높아지면 가장 1순위로 도입해야 할 강력한 억제기입니다.
